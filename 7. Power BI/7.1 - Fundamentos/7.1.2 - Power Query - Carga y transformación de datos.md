## 2. Qué es Power Query en profundidad

### Power Query es un pipeline, no una edición directa

Cuando limpias una columna en Power Query, **no estás modificando el archivo original**. El CSV de partida no cambia nunca. Power Query registra una secuencia de pasos de transformación y los aplica sobre el dato original cada vez que el modelo se refresca. Es como una función: `CSV_original → transformaciones → tabla limpia → modelo`.

Consecuencias prácticas que importan:

- Puedes deshacer cualquier paso sin consecuencias sobre el origen.
- Puedes auditar exactamente qué se ha hecho al dato, paso a paso.
- Si el CSV cambia mañana (llegan nuevas reservas), el pipeline se ejecuta de nuevo automáticamente sobre el nuevo dato sin que toques nada.
- Los errores de transformación no corrompen el origen, solo producen errores visibles en la tabla de destino.

Esta filosofía se llama **transformación no destructiva** y es uno de los pilares del trabajo riguroso con datos.

### El motor y el lenguaje M

Power Query usa un motor basado en el lenguaje **M** (Power Query Formula Language). Cada vez que haces clic en un botón de la interfaz, Power Query genera automáticamente una expresión M y la añade a la lista de pasos. Puedes ver ese código en cualquier momento con **Home > Advanced Editor**.

No necesitas escribir M a mano en la mayoría de transformaciones de hoy, pero en la sección de fechas sí lo harás una vez. En S12 lo trabajaremos en profundidad. Por ahora: cada vez que hagas una transformación, mira 10 segundos la barra de fórmulas y lee la expresión M generada. El hábito de conectar "clic en la UI → expresión M" te convierte en un usuario mucho más sólido.

## 3. Applied Steps: el concepto clave de la sesión

Abre `HotelesMediterraneum_S01.pbix` y entra al Editor de Power Query (**Home > Transform Data**).

En el panel derecho verás la sección **Applied Steps**. En este momento hay varios pasos automáticos generados al cargar el CSV:

- **Source**: conectar al archivo físico.
- **Promoted Headers**: usar la primera fila como cabeceras.
- **Changed Type**: intento automático de tipado de columnas.

Esta lista es el corazón de todo. El dato fluye de arriba a abajo: cada paso toma la salida del anterior como entrada.

### Propiedades que necesitas conocer

**Son navegables.** Haz clic en cualquier paso y verás el estado de la tabla en ese punto del pipeline. Puedes revisar el dato antes y después de cualquier transformación, no solo al final.

**Son eliminables.** Pulsa la X junto al nombre del paso. Si el paso que eliminas afectaba a pasos posteriores, Power Query te avisará.

**Son editables.** Haz doble clic en un paso para modificar sus parámetros.

**Generan código M.** Cada paso corresponde a una línea en el Advanced Editor.

## 4. Tipos de datos en Power Query

Antes de limpiar valores, conviene tener claro el sistema de tipos, porque muchas transformaciones de hoy implican conversiones.

| Tipo | Icono en cabecera | Descripción |
|---|---|---|
| Text | ABC | Cadenas de texto. |
| Whole Number | 123 | Enteros sin decimales. |
| Decimal Number | 1.2 | Números con decimales. |
| Fixed Decimal | $ | Moneda, 4 decimales fijos. Ideal para `revenue`. |
| Date | calendario | Solo fecha, sin hora. |
| Logical | True/False | Booleano. |
| Any | ? | Sin tipo. Power Query no hará validación. |

El icono en el ángulo superior izquierdo de cada cabecera indica el tipo actual. Cuando el tipo es incorrecto, las transformaciones y cálculos producen errores silenciosos o resultados absurdos.

Regla general: **ninguna columna debería salir de Power Query con tipo Any o con un tipo incorrecto**. Si hay ambigüedad, decide conscientemente y documéntalo en el nombre del Applied Step.

## 6. Parseo de fechas con formatos mezclados

Problema 2: las tres columnas de fecha (`booking_date`, `checkin_date`, `checkout_date`) están como texto porque mezclan tres formatos distintos: `2024-06-15`, `15/06/2024` y `5-7-24`. Resolvemos con dos enfoques: uno rápido para prototipo y uno riguroso para producción.

### 6.1 Enfoque rápido: Column from Examples

Power Query puede aprender el patrón de conversión a partir de ejemplos que tú le das, sin que escribas código.

Selecciona la columna `booking_date`. Ve a **Add Column > Column from Examples > From Selection**. Se abre un panel en la parte inferior con una columna nueva vacía a la derecha.

En la primera fila, escribe la fecha en formato ISO (`YYYY-MM-DD`) que corresponde al valor de esa fila. Pulsa Enter. Power Query intentará autocompletar el resto. Si acierta, pulsa **OK**. Si no, rellena un par de ejemplos más (uno de cada formato) hasta que aprenda el patrón.

> [CAPTURA 6.1 — Panel Column from Examples abierto mostrando la columna booking_date original (con formatos mezclados) y la columna nueva (con formato ISO uniforme) generada a partir de 2–3 ejemplos escritos.]

Resultado: columna nueva con todas las fechas en texto ISO uniforme. Ahora:

1. Elimina la columna original `booking_date` (clic derecho > Remove).
2. Renombra la nueva como `booking_date`.
3. Cambia el tipo a **Date** (clic en el icono del cabecero > Date).

Repite para `checkin_date` y `checkout_date`.

### 6.2 Enfoque riguroso: Custom Column con lógica M explícita

Column from Examples es rápido pero opaco. Si llega un cuarto formato no visto, puede fallar silenciosamente. El enfoque riguroso documenta explícitamente la lógica de detección de formato.

Ve a **Add Column > Custom Column**. Nombre: `booking_date_parsed`. Fórmula:

```
if Text.Contains([booking_date], "/")
then Date.FromText([booking_date], [Format="dd/MM/yyyy"])
else if Text.Length([booking_date]) <= 7
then Date.FromText(
    Text.PadStart(
        Text.BeforeDelimiter([booking_date], "-"),
    2, "0")
    & "-"
    & Text.PadStart(
        Text.AfterDelimiter(
            Text.BeforeDelimiter([booking_date], "-", 1),
        "-"),
    2, "0")
    & "-20"
    & Text.AfterDelimiter([booking_date], "-", 1),
    [Format="dd-MM-yyyy"]
)
else Date.FromText([booking_date], [Format="yyyy-MM-dd"])
```

Esta fórmula implementa la siguiente lógica de detección:

- Si el valor contiene `/` → formato europeo `DD/MM/YYYY`.
- Si el valor tiene 7 caracteres o menos → formato corto `D-M-YY`, reconstruido con ceros y prefijo `20` de año.
- En cualquier otro caso → formato ISO `YYYY-MM-DD`.

> [CAPTURA 6.2 — Diálogo Custom Column con el nombre booking_date_parsed y la fórmula M completa. El indicador de estado debe mostrar "No syntax errors have been detected".]

Cambia el tipo de la columna resultante a **Date**. Compara con la columna obtenida con Column from Examples: deberían ser idénticas. La diferencia está en la robustez: la versión M produce un error visible si llega un formato inesperado, en lugar de un resultado silenciosamente incorrecto.

**¿Cuándo usar cada enfoque?** Column from Examples para prototipos rápidos o cuando el origen es estable. Custom Column en M cuando el pipeline vaya a producción, el origen pueda cambiar, o necesites documentar explícitamente las reglas de negocio del parseo. En este bloque, para avanzar con comodidad en sesiones siguientes, trabaja con Column from Examples y conserva la fórmula M como referencia de consulta.

> [CAPTURA 6.3 — Vista Data del Editor de Power Query con las tres columnas de fecha tipadas como Date (icono calendario) y valores uniformes en todas las filas.]

## 7. Gestión de nulos con criterio analítico

Problemas 4 y 5: `meal_plan` (~5% nulos) y `customer_email` (~3% nulos). Antes de actuar, conviene pensar.

### El diagnóstico previo a la acción

Un nulo puede significar tres cosas distintas, y la acción correcta depende de cuál sea:

**Dato faltante por error de captura.** Debería tener valor pero no se registró. Acción apropiada: imputar la moda, o un valor neutro que distinga "no sabemos" de un valor real.

**Dato no aplicable.** El campo no tiene sentido para ese registro. Acción apropiada: imputar "Not Applicable" o dejar nulo con documentación explícita.

**Dato opcional.** El campo no es obligatorio. Acción apropiada: dejar nulo y gestionar en las medidas DAX con `ISBLANK()` cuando sea necesario.

Para HotelesMediterraneum adoptamos estas decisiones de negocio:

- **`meal_plan` nulo → imputar "Room Only"**: si no se registró régimen de comida, asumimos la opción más conservadora. Es la decisión que menos distorsiona los KPIs de revenue.
- **`customer_email` nulo → dejar nulo**: el email es un dato de contacto opcional. Imputarlo no tiene sentido analítico.

### Implementación

**Para `meal_plan`:** selecciona la columna. En la cinta **Transform > Replace Values**. En el diálogo, deja el campo "Value to Find" **vacío** (Power Query interpreta campo vacío como nulos) y en "Replace With" escribe `Room Only`.

> [CAPTURA 7.1 — Diálogo Replace Values con el campo "Value to Find" vacío y "Replace With" con "Room Only". Señalar que el campo vacío apunta a nulos.]

**Para `customer_email`:** no hacemos nada. Los nulos permanecen. Renombra el Applied Step (si añadiste alguno) como "customer_email: nulos deliberados, no imputar" para documentar la decisión.

Este nivel de documentación —dejar constancia de lo que NO se hace y por qué— es tan importante como documentar lo que sí se hace. En un pipeline de producción, la ausencia de un paso de imputación puede interpretarse como un olvido o como una decisión. El nombre del paso lo aclara.

## 8. Columna condicional: resolver la incoherencia `status` vs `is_cancelled`

Problema 7: hay filas con `status = "Confirmed"` e `is_cancelled = True` simultáneamente. Las dos columnas deberían ser redundantes pero no lo son en el ~2% de los casos.

**Decisión de negocio:** `status` es la fuente de verdad. Tiene tres valores expresivos (Confirmed, Cancelled, NoShow) y es la columna que llega al modelo. `is_cancelled` es una columna derivada que debemos calcular nosotros, no importar del origen.

### Paso 1: eliminar `is_cancelled` original

Clic derecho sobre la cabecera de `is_cancelled` > **Remove**. Aparece el paso "Removed Columns" en Applied Steps.

### Paso 2: recalcular `is_cancelled` correctamente

Ve a **Add Column > Conditional Column**. Se abre un diálogo visual:

- Nombre de la nueva columna: `is_cancelled`
- Condición: columna `status` | operador `equals` | valor `Cancelled`
- Output si verdad: `true`
- Output si falso: `false`

> [CAPTURA 8.1 — Diálogo Conditional Column configurado: columna "status", operador "equals", valor "Cancelled", output "true", else output "false".]

Pulsa OK. Cambia el tipo de la nueva columna a **Logical** (True/False).

Ahora `is_cancelled` es 100% consistente con `status` por construcción. Si en el futuro cambia el status de una reserva, `is_cancelled` se recalcula automáticamente en el próximo refresco.

Este patrón —eliminar una columna problemática del origen y reconstruirla de forma controlada— es uno de los más valiosos de Power Query en entornos de producción. Siempre es preferible una columna derivada correcta a una columna importada incoherente.

## 10. Lo que acabas de hacer sin saberlo

### El linaje de transformación

Has construido un **pipeline de linaje de datos** completo: cada Applied Step es una capa documentada que transforma el dato de un estado a otro. En el mundo profesional, el linaje (saber de dónde vienen los datos y qué transformaciones han sufrido) es un requisito de gobernanza. En el bloque de Gobierno del Dato hablabais de metadata y trazabilidad: este pipeline es exactamente eso aplicado a Power BI.

### Power Query como capa de calidad

Has implementado, sin nombrarlo así, un proceso de validación y remediación de calidad equivalente a lo que en producción haría un ETL más sofisticado. La lógica es la misma: detectar anomalías → decidir la regla de remediación → aplicar → validar resultado. La diferencia con Python/Spark es la escala y la flexibilidad programática, no el concepto.

### El equivalente en pandas

Si hubieras hecho estas mismas transformaciones en Python, el equivalente aproximado sería:

```python
# Limpieza de texto
df['channel'] = (df['channel'].str.strip()
                              .str.title()
                              .replace({'Booking.Com': 'Booking'}))
df['room_type'] = (df['room_type'].str.strip()
                                  .str.title()
                                  .replace({'Stndard': 'Standard'}))

# Parseo de fechas (simplificado)
df['booking_date'] = pd.to_datetime(df['booking_date'], dayfirst=True, errors='coerce')
df['checkin_date'] = pd.to_datetime(df['checkin_date'], dayfirst=True, errors='coerce')
df['checkout_date'] = pd.to_datetime(df['checkout_date'], dayfirst=True, errors='coerce')

# Nulos
df['meal_plan'] = df['meal_plan'].fillna('Room Only')

# Coherencia is_cancelled
df.drop(columns=['is_cancelled'], inplace=True)
df['is_cancelled'] = df['status'] == 'Cancelled'
```

Power Query ha hecho todo eso con clics y algunos pasos M puntuales. La ventaja es la trazabilidad y la integración con el modelo de datos. La desventaja es la flexibilidad para transformaciones muy complejas, donde Python sigue siendo más potente.

## 11. Errores comunes y troubleshooting

**"He eliminado un paso y los pasos siguientes dan error."**
Normal cuando existe dependencia entre pasos. Usa Ctrl+Z para deshacer, o revisa en qué paso aparece el primer triángulo rojo de error, haz clic en él para leer el mensaje, y reconstruye desde ahí.

**"Al usar Replace Values con campo vacío no me reemplaza los nulos."**
En algunas versiones de Power BI Desktop el campo vacío no equivale a nulo. En el diálogo Replace Values, busca la opción avanzada "Replace using special values > null". Alternativamente, usa Transform > Replace Values y en la ventana avanzada activa la casilla de nulos explícitamente.

**"Column from Examples no acierta el patrón de mis fechas."**
Rellena manualmente al menos un ejemplo de cada formato distinto que exista en la columna. Con dos o tres ejemplos por formato suele ser suficiente. Si sigue fallando, usa el enfoque riguroso de Custom Column con la fórmula M de la sección 6.2.

**"Tras Close & Apply el gráfico sigue mostrando 12 barras."**
Revisa: (1) ¿Guardaste el archivo? (2) Vuelve a Transform Data y abre el filtro desplegable de `channel`. Si ves más de 5 valores, hay un paso de limpieza que no funcionó. Edita ese paso o añade un Replace Values adicional para el valor residual.

**"El tipo de la columna de fecha queda como Date/Time en lugar de Date."**
Haz clic en el icono de la cabecera y selecciona explícitamente Date (no Date/Time). El componente horario (00:00:00) se descartará.

**"Quiero deshacer todos los pasos y empezar de cero."**
Cierra el Editor sin aplicar cambios (cierra la ventana o usa Close & Discard si aparece), luego en el panel de consultas (izquierda del Editor) haz clic derecho sobre la consulta y elige Delete. Recarga el CSV desde Get Data.

## 13. Ampliación opcional: los cuatro problemas restantes

Si has terminado los pasos anteriores y te sobra tiempo, o si quieres llegar a S03 con el dataset completamente limpio, estos son los cuatro problemas de calidad que quedaron fuera del horario principal.

### Extra 1: `nights` inconsistente con `checkout_date - checkin_date` (~4%)

La columna `nights` debería ser siempre igual a la diferencia en días entre `checkout_date` y `checkin_date`. Hay un pequeño porcentaje de filas donde no se cumple.

**Decisión de negocio:** las fechas son la fuente de verdad (son validadas por el sistema de reservas; `nights` es un campo libre susceptible a errores de entrada). Recalculamos `nights` a partir de las fechas.

**Implementación:** con las tres columnas de fecha ya tipadas como Date, ve a **Add Column > Custom Column**. Nombre: `nights_calculado`. Fórmula:

```
Duration.Days([checkout_date] - [checkin_date])
```

Cambia el tipo a Whole Number. Elimina la columna original `nights`. Renombra la nueva como `nights`.

### Extra 2: `children` con valores negativos (~0.5%)

Valores negativos en un campo de conteo de personas son imposibles. Origen probable: errores de entrada manual.

**Decisión de negocio:** interpretamos que el valor correcto es 0 (sin niños).

**Implementación:** Add Column > Custom Column. Nombre: `children_limpio`. Fórmula:

```
if [children] < 0 then 0 else [children]
```

Cambia el tipo a Whole Number. Elimina la original. Renombra.

### Extra 3: `revenue` con outliers extremos (~0.3%)

Hay reservas con revenue ~10 veces superior al esperado. Pueden ser errores de captura o reservas legítimas grandes (bloques para eventos).

**Decisión de negocio:** no imputamos ni eliminamos en Power Query porque no tenemos suficiente contexto. Marcamos con una bandera para que el analista pueda filtrarlos según el análisis.

**Implementación:** Add Column > Custom Column. Nombre: `revenue_flag`. Fórmula:

```
if [revenue] > 5000 then "Revisar" else "OK"
```

El umbral de 5.000 euros es orientativo; ajústalo según la distribución que veas en Column Profile. Usaremos esta columna en dashboards futuros para permitir al usuario ver el revenue con o sin outliers.

### Extra 4: espacios en blanco en `customer_name` (~3%)

Algunos nombres tienen espacios sobrantes al inicio o final.

**Implementación:** selecciona la columna `customer_name` y aplica **Transform > Format > Trim**. Un solo clic.
